# OUDS Flutter -  AI Agent Guidelines

This document provides context and guidelines for AI agents assisting with the OUDS Flutter (Orange Unified Design System for Flutter) project.

---

## 1. Code formatting

The source code is written in Dart and formatted with `dart format`. Linting is configured via `analysis_options.yaml` in each package using `package:flutter_lints/flutter.yaml`.

Before any commit, run:
```bash
dart format .
flutter analyze --no-pub
```

No `// ignore:` suppression is allowed unless strictly justified with an explanatory comment on the same line.

---

## 2. Architecture details

The repository is a multi-package Flutter workspace managed with `pub` (native Dart workspace, `pubspec.yaml` at root).

### 2.1 Packages

#### `ouds_core`

Contains all reusable UI components (widgets) provided by OUDS Flutter: buttons, switches, checkboxes, chips, tags, links, badges, navigation bars, top bars, form inputs, etc.

- Components are located in `ouds_core/lib/components/<name>/`
- Each component class is prefixed with `Ouds` (e.g. `OudsButton`, `OudsTag`)
- Every component reads visual values exclusively through the active theme tokens

#### `ouds_theme_contract`

Defines the `OudsThemeContract` abstract interface, the `OudsTheme` InheritedModel widget, semantic token abstractions and component token interfaces.

> ⚠️ Token interface files in this package are **generated by Tokenator** (Figma → Dart). Do not edit them manually.

#### `ouds_theme_orange`

Implements `OudsThemeContract` for all Orange brand products. Provides `OrangeTheme`.

#### `ouds_theme_orange_compact`

Implements `OudsThemeContract` for Orange products with tighter space and size constraints. Provides `OrangeCompactTheme`.

#### `ouds_theme_sosh`

Implements `OudsThemeContract` for all Sosh brand products. Provides `SoshTheme`.

#### `ouds_theme_wireframe`

Implements `OudsThemeContract` for prototyping and mockups. Provides `WireframeTheme`.

#### `ouds_global_raw_tokens`

Contains raw design token values (colors as hex, spacing as floats, typography sizes, etc.).

> ⚠️ All files in this package are **generated by Tokenator** (Figma → Dart). Never edit them manually.

#### `ouds_accessibility_plugin`

Native plugin (Android + iOS) providing platform-level accessibility features: high-contrast detection, system font scale. Used internally by `OudsCheckbox`, `OudsSwitch`, `OudsRadioButton`.

#### `app`

Demo application "Design System Toolbox" showcasing all components and tokens interactively. Every component must have a corresponding demo screen registered in `app/lib/ui/components/components.dart`.

---

### 2.2 Architecture guidelines

- Flutter / Material 3 is the default UI paradigm — embrace its declarative and reactive nature
- When creating a new component, try to match the equivalent Material 3 widget API as closely as possible
- Avoid unnecessary abstractions and over-engineering
- Focus on simplicity, clarity, and idiomatic Dart
- Organize by component — keep related code together
- Use extensions to organise large files
- Follow [Effective Dart](https://dart.dev/guides/language/effective-dart) naming conventions consistently
- Use `sealed class` for exhaustive type modelling (e.g. `OudsIconStatus`)
- Prefer `StatelessWidget` unless state is strictly necessary
- Use `const` constructors wherever possible

## 3. Development guidelines

### 3.1 Code style

- **Flutter / Material 3 best practices**: use modern declarative widget patterns — `StatelessWidget`, `const`, `InheritedModel`
- **Coding conventions**: follow [Effective Dart](https://dart.dev/guides/language/effective-dart) and `package:flutter_lints` rules configured in each `analysis_options.yaml`
- **Naming**:
  - Prefix all OUDS widgets with `Ouds` (e.g. `OudsButton`, `OudsTag`)
  - Token classes end with `Tokens` (e.g. `OudsButtonTokens`, `OudsColorTokens`)

### 3.2 File organization

| Type | Path |
|------|------|
| Component widgets | `ouds_core/lib/components/<name>/ouds_<name>.dart` |
| Token interfaces | `ouds_theme_contract/lib/theme/tokens/` |
| Theme implementations | `ouds_theme_orange/lib/`, `ouds_theme_sosh/lib/`, etc. |
| Demo screens | `app/lib/ui/components/<name>/` |
| Demo registry | `app/lib/ui/components/components.dart` |

## 5. Best practices

### 5.1 DO

- Write documentation in dartdoc format (`///`) for all public APIs, with at least one minimal code example
- Use Dart's type system for safety — prefer strong types, avoid `dynamic`
- Use `public` only when truly needed — prefer `private` (`_`) visibility
- Apply **Clean Code, DRY, SOLID** and **TDD** principles
- Use `const` constructors everywhere possible
- Use `StatelessWidget` when state is not needed
- Use `sealed class` / `enum` for exhaustive type modelling
- Split large widgets into smaller private sub-widgets or methods
- Use `final` fields in all widget classes
- Use `static` for utility methods that do not need instance context
- If a third-party dependency is added or updated, document it in `THIRD_PARTY.md`

### 5.2 DON'T

- Add abstraction layers without clear benefit
- Use `dynamic` or `Object` unless strictly necessary
- Perform heavy computation inside `build()` — move it to `initState`, a provider, or `compute()`
- Call `setState` from `initState` — use `addPostFrameCallback` if needed
- Mix business logic with UI code
- Use `print()` — use `debugPrint` or a proper logging package
- Hardcode colors, dimensions, or strings — always use tokens and localizations
- Edit or create token files in `ouds_global_raw_tokens`, `ouds_theme_contract`, `ouds_theme_orange`, `ouds_theme_orange_compact`, `ouds_theme_sosh`, or `ouds_theme_wireframe` — all token files in these packages are generated by Tokenator

## 8. Common Tasks

### 8.1 Adding a New Component

When adding new components:

1. Verify that component tokens generated by Tokenator are defined in `ouds_theme_contract` and implemented in all theme packages (`ouds_theme_orange`, `ouds_theme_sosh`, `ouds_theme_orange_compact`, `ouds_theme_wireframe`)
2. Create the widget in `ouds_core/lib/components/<name>/ouds_<name>.dart` — prefixed with `Ouds`, trying to match the equivalent Material 3 widget API as closely as possible
3. Consume all visual values exclusively through tokens via `OudsTheme.of(context)`
4. Ensure `Semantics` is properly implemented for accessibility support
5. Add dartdoc comments (`///`) on all public APIs with at least one minimal code example
6. Export the widget from `ouds_core`'s barrel file
7. Add widget tests in `ouds_core/test/components/<name>/`
8. Add a demo screen in `app/lib/ui/components/<name>/` with interactive controls showcasing all variants
9. Register the demo in `app/lib/ui/components/components.dart`

> ⛔ **NEVER create or edit token files in the following packages — all their content is generated by Tokenator (Figma → Dart) and must not be touched manually:**
> - `ouds_global_raw_tokens/` — raw design values (colors, spacing, typography, etc.)
> - `ouds_theme_contract/` — abstract token interfaces and component token contracts
> - `ouds_theme_orange/` — Orange brand token implementations
> - `ouds_theme_orange_compact/` — Orange Compact brand token implementations
> - `ouds_theme_sosh/` — Sosh brand token implementations
> - `ouds_theme_wireframe/` — Wireframe theme token implementations
>
> If a new component does **not** have a dedicated token file generated by Tokenator, do **not** create one. Instead, reuse existing tokens from `OudsTheme.of(context).spaceScheme(context)`, `OudsTheme.of(context).colorScheme(context)`, `OudsTheme.of(context).typographyTokens`, or `OudsTheme.of(context).componentsTokens(context)` to style the component. Never create token files manually.


### 8.2 Adding a New Theme

1. Create a new package (e.g. `ouds_theme_newbrand/`)
2. Implement `OudsThemeContract` from `ouds_theme_contract`
3. Provide concrete token values — raw token values must be generated by Tokenator
4. Add the new theme to the demo `app/` for visual testing
5. Test all existing components against the new theme in light and dark modes
6. Update documentation

### 8.3 Adding Accessibility & Semantics to a New Component

Every new component **must** expose proper accessibility semantics:

- Wrap the root interactive element in a `Semantics` widget with appropriate flags (`button`, `checked`, `toggled`, `label`, `hint`, `value`)
- Use `ExcludeSemantics` on purely decorative children so screen readers ignore them
- Use `MergeSemantics` to group related elements into a single focusable node
- Always provide a `semanticsLabel` parameter on components whose meaning is conveyed by color alone (e.g. `OudsBadge`)
- Always expose a `contentDescription` parameter on avatar/image components (e.g. `OudsAvatar`)
- Use `OudsLocalizations.of(context)` for all default accessibility strings — never hardcode theme

### 8.4 Modifying Existing Components

1. Check token definitions in `ouds_theme_contract` — do not edit generated files
2. Update the component implementation in `ouds_core/lib/components/<name>/`
3. Update demo screens in `app/lib/ui/components/<name>/` if the API changed
4. Update dartdoc comments if parameters changed

## 9. Testing

- Widget tests for component rendering and state in `ouds_core/test/`
- Interaction tests for user interactions such as taps, enabled/disabled state changes
- Run tests with `flutter test` in every modified package

### 9.1 Accessibility testing

- Provide `semanticsLabel` and `semanticsHint` on all interactive and informational widgets
- Ensure proper semantic roles — use `button`, `checked`, `toggled`, `label` flags in `Semantics` widgets
- Use `SemanticsController` in widget tests to verify semantic labels and flags are correctly set
- Support TalkBack (Android) and VoiceOver (iOS) — test on real devices or emulators
- Maintain sufficient color contrast — WCAG AA minimum (4.5:1 for normal text, 3:1 for large text)

## 11. Important Considerations

### 11.1 Multi-Brand Support

Always consider that changes affect **all brands**. Test components with:

- Orange theme
- Orange Compact theme
- Sosh theme
- Wireframe theme
- Light **and** dark modes

### 11.2 Accessibility

- Wrap interactive elements with `Semantics` widgets and provide `label`, `hint`, `value`, and `button`/`checked`/`toggled` flags
- Always set `semanticsLabel` on `OudsBadge` — color is the only visual differentiator
- Always set `contentDescription` on `OudsAvatar`
- Use `OudsLocalizations.of(context)` for all accessibility strings — never hardcode them
- Never lock text size — use OUDS typography tokens; never override `MediaQuery.textScalerOf`
- Do not lock the app in portrait mode — support landscape orientation
- Support high-contrast mode via `ouds_accessibility_plugin`
- Test with TalkBack (Android) and VoiceOver (iOS)
- Full guidelines: https://a11y-guidelines.orange.com/fr/mobile/

### 11.3 Backward Compatibility

- Minimum Android SDK 21, minimum iOS 14.0
- Avoid using recently added Flutter APIs without version guards
- Maintain stable public API surface — avoid breaking changes
- When a constructor is deprecated (e.g. `OudsTag()`, `OudsBadge()`, `OudsBadgeStatus`, `OudsTagStatus`), always use the modern API

### 11.4 Performance

- Avoid unnecessary `build()` calls — use `const`, `RepaintBoundary`, `ValueListenableBuilder`
- Offload heavy work to `compute()` or `Isolate`
- Free resources when the app goes to background (`AppLifecycleListener`)

### 11.5 Miscellaneous

- No hardcoded strings — use `OudsLocalizations` or `AppLocalizations`
- No hardcoded colors or dimensions — always use tokens
- Do not create or edit generated token files in `ouds_global_raw_tokens`, `ouds_theme_contract`, `ouds_theme_orange`, `ouds_theme_orange_compact`, `ouds_theme_sosh`, or `ouds_theme_wireframe`
